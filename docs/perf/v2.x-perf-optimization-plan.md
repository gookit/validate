# v2.x 性能优化实施计划（T1 / T3 / ValidResult 重构）

> **目的**：在已落地 T2（validator 免反射）+ Step2（map 懒分配）基础上，规划下一批性能优化的**详细实施方案**，供评审拍板。
> **状态**：设计/计划稿，未改业务代码。
> **日期**：2026-06-05
> **前置**：分析见 `gookit-vs-goplayground-gap-analysis.md`。当前 flat-valid 成功路径 **14 allocs / ~1687ns**（起点 23/2020），go-playground 为 **0 allocs / ~470ns**。

---

## 0. 现状与剩余差距的根

go-playground 0 分配 = **复用单实例 + struct 元信息缓存 + 取值不装箱 + 直接函数调用 + 只返回 error（不收集结果 map）**。

gookit 当前剩余 14 个 alloc 的三大来源：
1. **每次新建 `Validation` 实例**（即便 map 已懒分配，结构体本身 + 首次写入的少量 map 仍在）—— 受限于**结果生命周期**（`Validate()` 后还要从 `v` 读 `Errors`/`SafeData`/`BindStruct`），无法默认池化。← **ValidResult 重构解此根**
2. **每次新建 `StructData`**（FromStruct：StructData + `fieldNames`/`fieldValues` 两 map）。← **T1**
3. **取值装箱**：`StructData.TryGet` 返回 `fv.Interface()`，对非指针字段值装箱（~3.5 alloc/校验），下游校验器又常重新反射。← **T3**

本计划按 **价值×风险** 排序：**ValidResult（根因、收益最大、API 改动需慎）→ T1（中等、与池化协同）→ T3（侵入最大、最后做）**。

## 1. ValidResult 重构（核心）

### 1.1 问题

`Validation` 同时持有**配置**（rules/scenes/validators/data）与**结果**（`Errors`/`safeData`/`filteredData`）。结果在 `Validate()` 返回后才被读取：

```go
v := validate.Struct(data) // 新建实例(alloc)
ok := v.Validate()         // 校验, 结果写在 v 上
if !ok { use(v.Errors) }   // ← 结果生命周期超出 Validate
v.BindStruct(&out)         // ← 仍在用 v
```

→ `v` 必须存活到结果用完，**无法在 Validate 后归还池**。`Factory`+`Release()` 虽能复用，但要求调用方**手动管理生命周期**（用完显式 Release），心智负担重、易错，故只能 opt-in。

### 1.2 提案：把结果剥离到独立 `ValidResult`

新增一个**只含结果**的轻量结构，让配置实例在 `Validate` 后即可归还内部池，调用方只持有结果：

```go
// ValidResult 承载一次校验的结果, 与配置实例解耦, 可安全在 Validate 后长期持有。
type ValidResult struct {
    Errors       Errors         // 字段或方法二选一(见 1.4)
    safeData     map[string]any
    filteredData map[string]any
}

func (r *ValidResult) IsOK() bool
func (r *ValidResult) Fail() bool
func (r *ValidResult) Err() error                 // OneError, nil if ok
func (r *ValidResult) SafeData() map[string]any
func (r *ValidResult) Safe(key string) (any, bool)
func (r *ValidResult) BindStruct(ptr any) error   // 复用现有 Marshal+expandSafeData
func (r *ValidResult) BindSafeData(ptr any) error
// ...镜像现有结果读取子集
```

### 1.3 新入口（无状态、内部池化 —— 对标 go-playground）

提供"传入数据 → 返回结果"的**无状态**入口，内部用**包级默认 `sync.Pool`** 复用 `Validation`：

```go
// 概念签名(命名见 1.7 待决)
func Check(data any, scene ...string) *ValidResult      // struct/map 自适应
// 或方法式: func (v *Validation) ValidateR(scene ...string) *ValidResult
```

内部流程（关键在"移动而非拷贝"结果）：
```
1. v := defaultPool.Get()           // 复用实例(0 新建)
2. 配置 v（FromStruct/FromMap + 规则）  // 复用 typeCache 的不可变元信息
3. v.Validate()                      // 结果照常写入 v.Errors/v.safeData/v.filteredData
4. res := &ValidResult{              // 仅 1 次 alloc(结果壳)
     Errors:       v.Errors,         // ← 指针搬移(O(1)), 非拷贝
     safeData:     v.safeData,
     filteredData: v.filteredData,
   }
   v.Errors, v.safeData, v.filteredData = nil, nil, nil // v 交出结果所有权
5. v.Release()                       // 归还默认池(resetForReuse 已存在)
6. return res
```
- **搬移**结果 map 是 3 次指针赋值，O(1) 无拷贝；v 的 map 置 nil（懒分配下次写入再建）。
- 复用实例消除"新建 Validation"那批 alloc；唯一新增是 `ValidResult` 壳（1 alloc，且小）。
- 调用方拿到的 `res` 与 `v` 彻底无关，`v` 已回池 —— **生命周期问题根除，且无需手动 Release**。

### 1.4 API 细节决策

- **`Errors` 暴露**：现有 `v.Errors` 是**公开字段**。`ValidResult` 用字段 `Errors` 还是方法 `Errors()`？建议**字段**保持一致（`res.Errors`），但 `Errors` 与潜在方法同名冲突 → 折中：字段名 `Errors`，另给 `Err() error` / `IsOK()`。
- **结果读取子集**：只镜像"读结果"的方法（IsOK/Fail/Err/SafeData/Safe/SafeVal/Filtered/BindStruct/BindSafeData），**不**搬"配置"方法（AddRule/StringRule/Scene…，那些属于 `Validation`）。

### 1.5 API 清理（v2.0 破坏窗口仍开 → 直接干净移除，不弃用）

**版本前提**：`v2.0.0` tag 已**撤回**（2026-06-06），v1→v2 的破坏窗口仍开 —— 故**不背弃用包袱、一次做干净**，让结果出口唯一化。
> 提醒：Go proxy 对已抓取版本不可变缓存；撤回需确认 v2.0.0 未被 proxy 抓过，或在正式发布的 v2.0.0 里用 `retract` 处理早期误打的 tag。

**先厘清一个事实**：`v.Validate()` 有 `hasValidated` 守卫（`validating.go:78`），**同一个 v 不能重复 Validate**，Validate 后也不再加规则 —— "Validate 后用 v"**唯一真实行为是读结果**，**不存在"复用 v 这个校验器"的场景**（批量复用走 Factory.Release）。所以把结果剥离到 `ValidResult` 后，Validation 上的重结果方法没有存在必要。

**清理方案（结果出口唯一化）**：

- **Validation 仅保留轻量布尔面**（"过没过 + 看错误"）：`validate.Struct/Map/New` + `Validate() bool` + `v.Errors`（字段）+ `v.IsOK()/IsFail()` + `ValidateE/ValidateErr`。
- **从 Validation 移除数据提取方法**（不弃用、直接删）：`SafeData()/Safe()/SafeVal()/GetSafe()/Filtered()/FilteredData()/BindStruct()/BindSafeData()` —— 这些**只在 `ValidResult` 上**。
- **唯一结果对象 `ValidResult`**：要清洗后数据/绑定，必须经 `Check()/ValidateR()` 拿 `ValidResult`。

→ 职责清晰、一种写法：
- **纯校验**：`v := validate.Struct(d); if !v.Validate() { use(v.Errors) }`。
- **校验 + 取数据/绑定（推荐默认）**：`res := validate.Check(d); res.BindStruct(&out)` —— 池化、零生命周期负担。

**代价（churn）**：仓库内部用了 `v.SafeData()`/`v.BindStruct()` 等的**测试/示例需改走 `ValidResult`**；撤回/`retract` v2.0.0。这些一次性成本换长期干净的单一结果 API。

### 1.6 风险

- **内部改面**：移除 Validation 上的数据提取方法会波及一批内部测试/示例（`issues_x*_test.go` 里大量 `v.BindStruct`/`v.SafeData`）。需逐个改走 `ValidResult` 并跑全量；**这是本重构最大的 churn**。
- **结果搬移正确性**：必须确保搬移后 v 不再被读（已 Release）、且 v 的 map 置 nil 后懒分配链路正常。需 race + 复用污染测试覆盖。
- **默认池的并发**：包级 `sync.Pool` 本身并发安全；但要确保 `resetForReuse` 把一切残留清干净（已有，且 Factory 测试覆盖）。
- **`safeData`/`filteredData` 懒分配下可能为 nil**：搬移 nil 也 OK，`ValidResult` 方法对 nil map 读取安全。

### 1.7 决策（已定，2026-06-06）

1. **入口（两者都给）**：
   - `func (v *Validation) ValidateR(scene ...string) *ValidResult` —— primitive，与 `ValidateE`/`ValidateErr` 命名一致，作用于任意已配置实例（struct/map/builder），snapshot 结果 + Release（非池化 no-op）。
   - `func Check(structPtr any, scene ...string) *ValidResult` —— top-level、**文档推荐默认**、内部池化，对标 go-playground 的 `validate.Struct(s)`。map+程序化规则走 builder + `.ValidateR()`。
2. **`ValidResult.Errors` 用字段**（与现有公开 `v.Errors` 一致），另配 `IsOK()/Fail()/Err()` ergonomic 方法（不用同名方法 `Errors()`，会与字段冲突）。
3. 默认池：包级单池（足够）。
4. **新入口 `Check` 作为对外文档推荐默认**（本次为重构版本，主推无状态结果式 API）；旧 API 仍全兼容。

### 1.8 分期

- **P1a**：定义 `ValidResult` + 结果读取方法（复用现有 BindSafeData/SafeData 逻辑）。
- **P1b**：无状态入口 + 包级池 + 搬移&Release。
- **P1c**：基准对照（预期接近 Factory 复用水平：~11→更低）、文档与示例。

## 2. T1：池化 `StructData` / `fieldValues`

### 2.1 现状

`validate.Struct` → `FromStruct(s)` 每次新建 `StructData`（+ `fieldNames map[string]int8`、`fieldValues map[string]reflect.Value` 两 map）。类型**元信息**已由 `typeCache`（`cache.go`）缓存（不可变、跨调用复用）；`StructData` 是**每次实例**的值持有者。分析实测：连同 instantiate ≈ 7.5 alloc。

### 2.2 方案

1. **`fieldValues` 懒分配**（与 Step2 同思路）：仅在 `TryGet` 真正缓存字段值时才 make。多数小 struct 校验少量字段，省 1 alloc。
2. **池化 `StructData`**：在 Factory（及 ValidResult 的默认池路径）里把 `StructData` 也纳入复用 —— `get()` 时取一个 reset 过的 StructData，`Release()` 时 `clear(fieldNames)`/`clear(fieldValues)` 并解绑 `value`/`valueTyp`。
   - 注意：`StructData` 持有源对象的 `reflect.Value`（`value`）—— reset 必须解绑，避免持有上一个源对象（内存泄漏 + 污染）。
   - `fieldNames` 在 ValidResult/Factory 路径下由 `createInto` 重建；确认 reset 后重建语义正确。

### 2.3 风险/范围
- 中等。StructData 字段较多，reset 要全。靠 Factory 既有的"复用污染"测试 + race 兜底。
- 与 ValidResult 协同：ValidResult 的默认池可同时池化 Validation + StructData，收益叠加。

### 2.4 分期
- **P2a**：`fieldValues` 懒分配（小、独立、可先做）。
- **P2b**：StructData 池化（依赖池基础设施，建议跟 ValidResult 的池一起做）。

## 3. T3：去 `TryGet` 的 `fv.Interface()` 装箱取值

### 3.1 现状

`StructData.TryGet`（`data_source.go`）对字段值统一 `fv.Interface()` 返回 `any` —— 对非指针标量值**装箱**（堆分配），下游 `valueValidate` 又常把它重新 `reflect.ValueOf(val)`。分析实测 ~3.5 alloc/校验来自这里。go-playground 全程传 `reflect.Value`、按需取，不装箱。

### 3.2 方案（侵入最大，需专项）

核心：**让取值/校验链路端到端携带 `reflect.Value`，只在校验器真正需要 `any` 时才装箱**。
- 已有载体 `fieldval.FieldValue`（`valueValidate` 里 `fv := fieldval.New(field, val)`）—— 扩展它**同时携带 reflect.Value**，避免下游 `reflect.ValueOf(val)` 二次反射。
- `TryGet` 增加一个"返回 reflect.Value 而不装箱"的内部路径（如 `tryGetRV`），供 valueValidate 在能用 reflect.Value 的分支直接消费；只有走到必须 `any` 的内建/自定义校验器入参时才 `Interface()`。
- 对 switch 里**已免反射的标量校验器**（T2 成果），很多其实需要 string/标量 —— 可结合 `valToString` 直接从 reflect.Value 取，跳过中间 `any`。

### 3.3 风险/范围
- **最大**。触及取值与校验分发的核心数据流，边界多（指针/自定义类型/wildcard 切片/custom validator 的 `any` 入参/safeData 存的也是 `any`）。
- **必须**：专项设计子稿 + 充分 golden（逐字节）+ race + 全量；分小步合入，每步可回滚。

### 3.4 分期
- **P3a**：先出 T3 专项设计子稿（数据流改造点逐一列举 + 每点的 golden 风险）。
- **P3b**：分步实现（先 struct 源标量路径，再 map/json，再 custom）。
- 建议 **ValidResult + T1 落地、收益兑现后再评估是否值得做 T3**（性价比最低、风险最高）。

## 5. 实施路线图（建议顺序与验收）

| 步 | 内容 | 风险 | 验收 |
|---|---|---|---|
| **S1** ✅ | T1-P2a：`fieldValues` ~~懒分配~~ → **去掉 value 缓存写入**（实测见下） | 低 | golden×3 + 全量 + race ✅；基准 **−2 allocs / ~10% 提速** |
| **S2** ✅ | ValidResult P1a/P1b/P1c（结构+无状态入口+池+搬移+删旧 API） | 中（API） | golden×3 + 全量 + race + 新入口专项测试 + 复用污染/并发 ✅；`Check` 池化 **12→10 allocs**（见 §5.2） |
| **S3** ✅ | T1-P2b：StructData 池化（复用 `v.sd`，并入 Factory/Check 池） | 中 | 复用污染(跨类型) + race ✅；`Check` **10→7 allocs**（见 §5.3） |
| **S3.1** ✅ | instantiateStatic 复用 rules 切片（memprofile 制导） | 低 | golden×3 + race ✅；`Check` **7→6 allocs**（见 §5.4） |
| **S4** | T3：~~先出专项设计子稿，评审后分步实现~~ → **P3a 子稿已出，结论建议不实施**（见 `t3-deboxing-design.md` / §5.5） | 高 | —（评估后关闭，待用户最终决策） |

每步：conventional commit、`TestIssue`/`TestPerf` 类测试不留红、文档同步、改前后基准数字入文档。

### 5.1 S1 实施结论（2026-06-06，已落地）

**字面计划（fieldValues 懒分配）对真实校验零收益**：任何带规则的字段在首次 `TryGet` 即被缓存，map 照样按需分配，基准纹丝不动（14/34/17）。

**根因发现**：`StructData.fieldValues` 这个 *value 缓存本身是负优化*——`TryGet` 末尾无论是否命中缓存都要 `fv.Interface()` 装箱（真正的 alloc），而顶层字段走 `FieldByIndex(cached index)` 本就免分配；缓存只省下"廉价的 reflect 查找"，却付出 map + entries 的分配。**去掉 `TryGet` 的缓存写入**（保留 `Set()` 自己的写 + `TryGet` 对 `Set` 写入的读）：

| 基准 | 改前 | 改后(B) | Δ |
|---|---|---|---|
| FlatValid | 14 allocs / 1690ns / 1467B | **12 / 1510 / 1057** | −2 allocs / −10.6% ns |
| FlatInvalid | 34 / 2348 / 3460 | **32 / 2196 / 3060** | −2 allocs / −6.5% ns |
| NestedValid | 17 / 1787 / 1544 | **15 / 1598 / 1144** | −2 allocs / −10.6% ns |

唯一理论回退：同一**子结构**字段被多条规则反复读时重复 `strings.Split`（实测 2~3 规则的嵌套仍净赢）。若后续需消除该点，可把 sub-path 导航改为不分配（手动按 `.` 游走），见 S4/T3 协同。`ensureFieldValues` 懒分配 helper 保留（仅 `Set` 写入时用）。

### 5.2 S2 实施结论（2026-06-06，已落地）

分四个绿色提交：**P1a**（`result.go`：`ValidResult` + 读取方法 + `expandSafeData` 提自由函数）→ **P1b**（`Validation.ValidateR` 结果指针搬移 + 顶层 `Check` 走包级 `defaultFactory` 池 + `result_test.go`）→ **P2**（7 个测试文件 + 2 个示例改走 `ValidResult`）→ **P3**（从 `Validation` 删除 8 个数据提取方法，结果出口唯一化）。

**基准（_examples/bench-vs-goplayground，flat-valid 成功路径）**：

| 入口 | ns/op | B/op | allocs/op | 说明 |
|---|---|---|---|---|
| `Struct(&u).Validate()`（非池化） | 1542 | 1059 | 12 | S1 后基线，不变 |
| **`Check(&u)`（池化，推荐默认）** | **1528** | **848** | **10** | 实例池化省 −2 allocs（含 `ValidResult` 壳 +1） |
| go-playground（复用单实例） | 480 | 0 | 0 | 参照 |

**为何只到 10 而非计划估的 5–7**：实例池化消除了"新建 Validation"那批，但 `Check` 仍每次 `FromStruct` 新建 **StructData + fieldNames map**（≈ S3 目标）、TryGet 仍 `fv.Interface()` 装箱 ×N（≈ T3 目标）、safeData map + `ValidResult` 壳。计划估值把 S3 的 StructData 池化收益提前并入了，故 S2 单独兑现为 12→10；**S3 接力 StructData 池化**预期再下探到 ~5–7。

**API 收紧（破坏性）**：`Validation` 仅留布尔/错误面（`Validate() bool` / `Errors` / `IsOK/IsFail/IsSuccess` / `ValidateE/ValidateErr/ValidateData`）。数据/绑定全部移到 `ValidResult`，经 `Check()`（struct，推荐）或 `v.ValidateR()`（map/builder/任意已配置实例）获取。`expandSafeData` 自由函数保留供 `ValidResult.BindSafeData`。

### 5.3 S3 实施结论（2026-06-06，已落地）

`Check` 在 S2 后剩 10 allocs，主要来源是每次 `FromStruct` 新建的 **StructData + fieldNames map**。本步让池化路径复用一个挂在 `*Validation` 上的 `StructData`（`v.sd`）：

- `Validation` 加 `sd *StructData`（默认 New/Struct/Map 路径恒为 nil，零影响）。
- `StructData.reset()`：解绑 `src/value/valueTyp/meta` + 清 tag + `clear(fieldNames/fieldValues)`，**保留 map 分配**（复用的关键）；对 nil map 安全。
- `FromStruct` 抽出 `(d *StructData) fromStruct(s)`（先 `reset()` 再填充）；`FromStruct` 对外行为逐字节不变。
- `Factory.Struct` 复用 `v.sd`（首次 lazy 建）；`resetForReuse` 调 `v.sd.reset()` —— 既清污染又防"池中实例钉住上个源对象"的泄漏。

| 入口 | ns/op | B/op | allocs/op |
|---|---|---|---|
| `Check`（S2 后） | 1528 | 848 | 10 |
| **`Check`（S3 后）** | **1420** | **472** | **7** |
| 非池化 `Struct().Validate()` | 1527 | ~1074 | 12（默认路径不变） |

**−3 allocs / B/op 近腰斩**。落到计划估的 ~5–7 区间。跨类型复用专项用例 `TestCheck_poolReuse_mixedTypes` + 并发混类型 `TestCheck_concurrent_mixedTypes` 守护正确性。

### 5.4 S3.1 实施结论（2026-06-06，已落地）

对 `Check`（7 allocs）做了 memprofile（`tmp/check_mem.prof`），按 alloc_objects 拆出 7 个分配的确切来源：

| 来源 | 个数 | 性质 |
|---|---|---|
| `TryGet` 的 `fv.Interface()` 装箱（`reflect.packEface`） | ~3 | T3/S4，高风险 |
| safeData map（`ensureSafeData` 头 + `applyField:247` 插入桶） | ~2 | **必需输出**（搬移进 ValidResult，调用方持有），不可池化 |
| `ValidResult` 壳（`ValidateR`） | 1 | **必需**（结果对象本身） |
| `instantiateStatic:226` `v.rules = make(...)` | 1 | **可低风险摘除** |

唯一可低风险摘除的非装箱分配就是 rules 切片 make。改为 cap 足够时复用实例自有 backing array（`v.rules`/`v.filterRules`；`resetForReuse` 已 `[:0]`），默认路径 cap=0 仍 make、行为逐字节不变。

**基准：`Check` 7→6 allocs / 1420→1365 ns / 472→405 B**（`614bc37`）。至此池化路径的"非装箱、非必需输出"分配已摘净；**剩余 6 = 2(safeData) + 1(ValidResult 壳) + 3(装箱)**，再往下只剩 T3 去装箱（S4，高风险）与"不收集 safeData"这类功能取舍。

### 5.5 S4 / T3 评估结论（2026-06-07，专项子稿 `t3-deboxing-design.md`）

P3a 子稿已出，**结论：去装箱方向收益≈0，建议不实施**。核心论证（详见子稿 §4）：

> `Check`/`ValidateR` 通过的每个字段都要存入 `safeData map[string]any`，存入时**必须**装箱成 `any`；而 `tryGet` 先读 `safeData` 命中即返回，使"同字段多规则只装箱一次"——即当前已是 ≈1 装箱/已存字段，且**该装箱被 safeData 复用、是输出的固有成本**。把校验链改成端到端 `reflect.Value`（需重写 ~70 个 `any`-原生校验器 + 底层比较/取串，golden 风险极高），通过的字段**仍要为 safeData 装箱一次** → 净收益≈0；只有"取值但既不入 safeData 也不需字符串"的错误/跳过路径字段能省 1–2 alloc。

**真正的杠杆是"不收集 safeData"**（opt-in `CheckErr(s) error`，跳过 safeData/filteredData/ValidResult），已用抛弃式 spike 实测（2026-06-07，代码已回退，报告 `tmp/spike-checkerr-report.md`，详见子稿 §6.1）：

| 变体 | allocs | ns | B/op |
|---|---:|---:|---:|
| `Check`（基线） | 6 | 1389 | 405 |
| `CheckErr` 无去重 | **8 ⚠️** | 1428 | 105 |
| `CheckErr` + 1槽装箱去重 | **3** | 1175 | 40 |

**实测证实**：单跳过 safeData 反而 6→8（丢失装箱去重，多规则字段重复装箱）；**必须连同"连续同字段取值去重"一起做**才降到 3 allocs（−50%）。即 `CheckErr` 形态 C 可行且有效，但**是产品 API 决策**（opt-in 快速过/败入口 + 维护去重/zero 处理成本），与 T3"去装箱"是两回事。**T3 仍关闭；`CheckErr` 已正式化落地（2026-06-07，valid 实测 3 allocs，详见 §9 与 `checkerr-impl-plan.md`）。**
