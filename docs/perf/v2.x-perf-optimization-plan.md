# v2.x 性能优化实施计划（T1 / T3 / ValidResult 重构）

> **目的**：在已落地 T2（validator 免反射）+ Step2（map 懒分配）基础上，规划下一批性能优化的**详细实施方案**，供评审拍板。
> **状态**：设计/计划稿，未改业务代码。
> **日期**：2026-06-05
> **前置**：分析见 `gookit-vs-goplayground-gap-analysis.md`。当前 flat-valid 成功路径 **14 allocs / ~1687ns**（起点 23/2020），go-playground 为 **0 allocs / ~470ns**。

---

## 0. 现状与剩余差距的根

go-playground 0 分配 = **复用单实例 + struct 元信息缓存 + 取值不装箱 + 直接函数调用 + 只返回 error（不收集结果 map）**。

gookit 当前剩余 14 个 alloc 的三大来源：
1. **每次新建 `Validation` 实例**（即便 map 已懒分配，结构体本身 + 首次写入的少量 map 仍在）—— 受限于**结果生命周期**（`Validate()` 后还要从 `v` 读 `Errors`/`SafeData`/`BindStruct`），无法默认池化。← **ValidResult 重构解此根**
2. **每次新建 `StructData`**（FromStruct：StructData + `fieldNames`/`fieldValues` 两 map）。← **T1**
3. **取值装箱**：`StructData.TryGet` 返回 `fv.Interface()`，对非指针字段值装箱（~3.5 alloc/校验），下游校验器又常重新反射。← **T3**

本计划按 **价值×风险** 排序：**ValidResult（根因、收益最大、API 改动需慎）→ T1（中等、与池化协同）→ T3（侵入最大、最后做）**。

---

## 1. ValidResult 重构（核心）

### 1.1 问题

`Validation` 同时持有**配置**（rules/scenes/validators/data）与**结果**（`Errors`/`safeData`/`filteredData`）。结果在 `Validate()` 返回后才被读取：

```go
v := validate.Struct(data) // 新建实例(alloc)
ok := v.Validate()         // 校验, 结果写在 v 上
if !ok { use(v.Errors) }   // ← 结果生命周期超出 Validate
v.BindStruct(&out)         // ← 仍在用 v
```

→ `v` 必须存活到结果用完，**无法在 Validate 后归还池**。`Factory`+`Release()` 虽能复用，但要求调用方**手动管理生命周期**（用完显式 Release），心智负担重、易错，故只能 opt-in。

### 1.2 提案：把结果剥离到独立 `ValidResult`

新增一个**只含结果**的轻量结构，让配置实例在 `Validate` 后即可归还内部池，调用方只持有结果：

```go
// ValidResult 承载一次校验的结果, 与配置实例解耦, 可安全在 Validate 后长期持有。
type ValidResult struct {
    Errors       Errors         // 字段或方法二选一(见 1.4)
    safeData     map[string]any
    filteredData map[string]any
}

func (r *ValidResult) IsOK() bool
func (r *ValidResult) Fail() bool
func (r *ValidResult) Err() error                 // OneError, nil if ok
func (r *ValidResult) SafeData() map[string]any
func (r *ValidResult) Safe(key string) (any, bool)
func (r *ValidResult) BindStruct(ptr any) error   // 复用现有 Marshal+expandSafeData
func (r *ValidResult) BindSafeData(ptr any) error
// ...镜像现有结果读取子集
```

### 1.3 新入口（无状态、内部池化 —— 对标 go-playground）

提供"传入数据 → 返回结果"的**无状态**入口，内部用**包级默认 `sync.Pool`** 复用 `Validation`：

```go
// 概念签名(命名见 1.7 待决)
func Check(data any, scene ...string) *ValidResult      // struct/map 自适应
// 或方法式: func (v *Validation) ValidateR(scene ...string) *ValidResult
```

内部流程（关键在"移动而非拷贝"结果）：
```
1. v := defaultPool.Get()           // 复用实例(0 新建)
2. 配置 v（FromStruct/FromMap + 规则）  // 复用 typeCache 的不可变元信息
3. v.Validate()                      // 结果照常写入 v.Errors/v.safeData/v.filteredData
4. res := &ValidResult{              // 仅 1 次 alloc(结果壳)
     Errors:       v.Errors,         // ← 指针搬移(O(1)), 非拷贝
     safeData:     v.safeData,
     filteredData: v.filteredData,
   }
   v.Errors, v.safeData, v.filteredData = nil, nil, nil // v 交出结果所有权
5. v.Release()                       // 归还默认池(resetForReuse 已存在)
6. return res
```
- **搬移**结果 map 是 3 次指针赋值，O(1) 无拷贝；v 的 map 置 nil（懒分配下次写入再建）。
- 复用实例消除"新建 Validation"那批 alloc；唯一新增是 `ValidResult` 壳（1 alloc，且小）。
- 调用方拿到的 `res` 与 `v` 彻底无关，`v` 已回池 —— **生命周期问题根除，且无需手动 Release**。

### 1.4 API 细节决策

- **`Errors` 暴露**：现有 `v.Errors` 是**公开字段**。`ValidResult` 用字段 `Errors` 还是方法 `Errors()`？建议**字段**保持一致（`res.Errors`），但 `Errors` 与潜在方法同名冲突 → 折中：字段名 `Errors`，另给 `Err() error` / `IsOK()`。
- **结果读取子集**：只镜像"读结果"的方法（IsOK/Fail/Err/SafeData/Safe/SafeVal/Filtered/BindStruct/BindSafeData），**不**搬"配置"方法（AddRule/StringRule/Scene…，那些属于 `Validation`）。

### 1.5 向后兼容（最重要约束）

**现有 API 一律不动、不破坏**：`validate.Struct/Map/New` + `Validate() bool` + `v.Errors`/`v.SafeData()`/`v.BindStruct()`/`ValidateE`… 全部保持现状语义（实例持有结果、不池化）。`ValidResult` 与新入口是**纯增量 opt-in**。

→ 两条路并存：
- **旧**：`v := validate.Struct(d); v.Validate(); v.Errors` —— 兼容、不池化（适合需要继续配置/复用 v 的场景）。
- **新**：`res := validate.Check(d); res.IsOK()` —— 池化、零生命周期负担（适合一次性校验、批量、高频）。

### 1.6 风险

- **API 表面扩张**："做同一件事有两种写法"。需文档清楚定位（新入口用于一次性/高频；旧 API 用于需继续操作 v 的场景）。
- **结果搬移正确性**：必须确保搬移后 v 不再被读（已 Release）、且 v 的 map 置 nil 后懒分配链路正常。需 race + 复用污染测试覆盖。
- **默认池的并发**：包级 `sync.Pool` 本身并发安全；但要确保 `resetForReuse` 把一切残留清干净（已有，且 Factory 测试覆盖）。
- **`safeData`/`filteredData` 懒分配下可能为 nil**：搬移 nil 也 OK，`ValidResult` 方法对 nil map 读取安全。

### 1.7 决策（已定，2026-06-06）

1. **入口（两者都给）**：
   - `func (v *Validation) ValidateR(scene ...string) *ValidResult` —— primitive，与 `ValidateE`/`ValidateErr` 命名一致，作用于任意已配置实例（struct/map/builder），snapshot 结果 + Release（非池化 no-op）。
   - `func Check(structPtr any, scene ...string) *ValidResult` —— top-level、**文档推荐默认**、内部池化，对标 go-playground 的 `validate.Struct(s)`。map+程序化规则走 builder + `.ValidateR()`。
2. **`ValidResult.Errors` 用字段**（与现有公开 `v.Errors` 一致），另配 `IsOK()/Fail()/Err()` ergonomic 方法（不用同名方法 `Errors()`，会与字段冲突）。
3. 默认池：包级单池（足够）。
4. **新入口 `Check` 作为对外文档推荐默认**（本次为重构版本，主推无状态结果式 API）；旧 API 仍全兼容。

### 1.8 分期

- **P1a**：定义 `ValidResult` + 结果读取方法（复用现有 BindSafeData/SafeData 逻辑）。
- **P1b**：无状态入口 + 包级池 + 搬移&Release。
- **P1c**：基准对照（预期接近 Factory 复用水平：~11→更低）、文档与示例。

## 2. T1：池化 `StructData` / `fieldValues`

### 2.1 现状

`validate.Struct` → `FromStruct(s)` 每次新建 `StructData`（+ `fieldNames map[string]int8`、`fieldValues map[string]reflect.Value` 两 map）。类型**元信息**已由 `typeCache`（`cache.go`）缓存（不可变、跨调用复用）；`StructData` 是**每次实例**的值持有者。分析实测：连同 instantiate ≈ 7.5 alloc。

### 2.2 方案

1. **`fieldValues` 懒分配**（与 Step2 同思路）：仅在 `TryGet` 真正缓存字段值时才 make。多数小 struct 校验少量字段，省 1 alloc。
2. **池化 `StructData`**：在 Factory（及 ValidResult 的默认池路径）里把 `StructData` 也纳入复用 —— `get()` 时取一个 reset 过的 StructData，`Release()` 时 `clear(fieldNames)`/`clear(fieldValues)` 并解绑 `value`/`valueTyp`。
   - 注意：`StructData` 持有源对象的 `reflect.Value`（`value`）—— reset 必须解绑，避免持有上一个源对象（内存泄漏 + 污染）。
   - `fieldNames` 在 ValidResult/Factory 路径下由 `createInto` 重建；确认 reset 后重建语义正确。

### 2.3 风险/范围
- 中等。StructData 字段较多，reset 要全。靠 Factory 既有的"复用污染"测试 + race 兜底。
- 与 ValidResult 协同：ValidResult 的默认池可同时池化 Validation + StructData，收益叠加。

### 2.4 分期
- **P2a**：`fieldValues` 懒分配（小、独立、可先做）。
- **P2b**：StructData 池化（依赖池基础设施，建议跟 ValidResult 的池一起做）。

## 3. T3：去 `TryGet` 的 `fv.Interface()` 装箱取值

### 3.1 现状

`StructData.TryGet`（`data_source.go`）对字段值统一 `fv.Interface()` 返回 `any` —— 对非指针标量值**装箱**（堆分配），下游 `valueValidate` 又常把它重新 `reflect.ValueOf(val)`。分析实测 ~3.5 alloc/校验来自这里。go-playground 全程传 `reflect.Value`、按需取，不装箱。

### 3.2 方案（侵入最大，需专项）

核心：**让取值/校验链路端到端携带 `reflect.Value`，只在校验器真正需要 `any` 时才装箱**。
- 已有载体 `fieldval.FieldValue`（`valueValidate` 里 `fv := fieldval.New(field, val)`）—— 扩展它**同时携带 reflect.Value**，避免下游 `reflect.ValueOf(val)` 二次反射。
- `TryGet` 增加一个"返回 reflect.Value 而不装箱"的内部路径（如 `tryGetRV`），供 valueValidate 在能用 reflect.Value 的分支直接消费；只有走到必须 `any` 的内建/自定义校验器入参时才 `Interface()`。
- 对 switch 里**已免反射的标量校验器**（T2 成果），很多其实需要 string/标量 —— 可结合 `valToString` 直接从 reflect.Value 取，跳过中间 `any`。

### 3.3 风险/范围
- **最大**。触及取值与校验分发的核心数据流，边界多（指针/自定义类型/wildcard 切片/custom validator 的 `any` 入参/safeData 存的也是 `any`）。
- **必须**：专项设计子稿 + 充分 golden（逐字节）+ race + 全量；分小步合入，每步可回滚。

### 3.4 分期
- **P3a**：先出 T3 专项设计子稿（数据流改造点逐一列举 + 每点的 golden 风险）。
- **P3b**：分步实现（先 struct 源标量路径，再 map/json，再 custom）。
- 建议 **ValidResult + T1 落地、收益兑现后再评估是否值得做 T3**（性价比最低、风险最高）。

## 4. 三者协同的预估

| 阶段 | flat-valid allocs（预估） | 说明 |
|---|---|---|
| 现状（T2+Step2 后） | 14 | 已落地 |
| +ValidResult（实例池化） | ~5–7 | 消除"新建 Validation"批；剩 StructData + 装箱 |
| +T1（StructData 池化） | ~3–5 | 再消除 StructData 批 |
| +T3（去装箱） | ~1–2 | 逼近 go-playground（其 0 靠极致复用+无结果收集） |

> 注：gookit 永远会比 go-playground 多至少"结果壳/收集"那点开销（它**收集 safeData 供 BindStruct**，是功能取舍）。目标是把可控差距压到 1–2 alloc 量级，而非追平 0。

## 5. 实施路线图（建议顺序与验收）

| 步 | 内容 | 风险 | 验收 |
|---|---|---|---|
| **S1** | T1-P2a：`fieldValues` 懒分配 | 低 | golden×3 + 全量 + race；基准 −1 alloc |
| **S2** | ValidResult P1a/P1b/P1c（结构+无状态入口+池+搬移） | 中（API） | golden×3 + 全量 + race + 新入口专项测试 + 复用污染/并发；基准接近 Factory 水平 |
| **S3** | T1-P2b：StructData 池化（并入 ValidResult 的池） | 中 | 复用污染 + race；基准再 −2~3 alloc |
| **S4** | T3：先出专项设计子稿，评审后分步实现 | 高 | 每步 golden 逐字节 + race；可回滚 |

每步：conventional commit、`TestIssue`/`TestPerf` 类测试不留红、文档同步、改前后基准数字入文档。

## 6. 风险与回滚
- ValidResult 是**增量**，不破坏旧 API；若新入口设计不满意可不发布（仅内部）。
- T1/T3 改业务热路径 —— **golden 逐字节是行为不变的硬证明**，每步独立提交可单独 revert。
- 所有步骤**默认不改对外行为**（ValidResult 是新增 API，非替换）。

## 7. 决策汇总（已定，2026-06-06）
1. ✅ 入口两者都给：`Check`（top-level，推荐默认，池化）+ `v.ValidateR()`（primitive）。
2. ✅ `ValidResult.Errors` 用**字段** + `IsOK()/Fail()/Err()` 方法。
3. ✅ **T3 延后**（路线图 S4 保留，S2/S3 收益兑现后再评估）。
4. ✅ 新入口 `Check` 作为**对外文档推荐默认**；旧 API 全兼容。
