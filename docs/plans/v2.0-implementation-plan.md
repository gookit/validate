## 0. 总览

v1.6.0 已完成**保持公共 API 兼容**的内部性能重构（扁平 struct -78% 耗时 / -74% 分配）。
v2.0.0 在此之上做**破坏性**精简，核心五件事：

1. **验证器统一 `reflect.Value` 入参**：内置 70+ 验证器内部实现统一收 `reflect.Value`，删除按调用转换的 `convertArgsType` 整套机器（v1.6.0 的最大遗留妥协）。
2. **`DataFace` 扩展 + 消除源头 `Interface()` 装箱**：`TryGet` 返回携带 `reflect.Value` 的载体，落地设计 §4.3 完整版（v1.6.0 因属破坏性而搁置的最后一跳）。
3. **注册自定义类型**：开放 `can register custom type` 能力。
4. **清理历史包袱**：删 deprecated（`ValueLen` 等）、升 Go 1.21、模块升 `/v2`。
5. **opt-in Factory/Pool**：落地 v1.6.0 延后的 `Struct/Map` 池化工厂。

**破坏性边界（明确告知用户）**：
- 直接调用内置验证器函数（如 `validate.Gt(x, int64(5))`）的代码需改签名。
- 自定义 `DataFace` 实现者需实现扩展后的接口。
- import 路径需带 `/v2`；最低 Go 版本升至 1.21。

**红线（每阶段都必须满足）**：
- **行为零回归**：v1.6.0 的 `regression_compat_test.go` 黄金网 + 全量测试（`issues_test.go`/`issues_x2_test.go`/`validation_test.go`/`validators_test.go`/`data_source_test.go`/`morecase_test.go`）须全绿。
  - ⚠️ 例外：V3（验证器签名）会**主动改变** golden 的 `%#v` 形态（args 类型语义变了）——届时**同步更新 golden 期望值**并在提交说明中标注，而非视作回归。
- **性能不回退**：以 v1.6.0 终态（`docs/plans/p6-final.txt`）为基线；理想是 V3/V4 进一步降 allocs。
- 每个**破坏性变更**当场记入 §5 破坏性清单（喂给最终迁移指南）。

> 📌 设计 §6.0 原话："v2.0.0 的破坏性内容待 1.6.0 稳定发布后另立设计文档"。本次按用户决策**不另写设计文档**，范围以本计划 §3 为准；其中 V5（注册自定义类型）API 形态尚未细化，执行前需先定（见该阶段「前置决策」）。

## 1. 工作约定

### 1.1 构建 / 测试 / 基准命令
```bash
go build ./...
go vet ./...
go test ./...                              # 全量回归
go test -race ./...                        # 并发安全（V4/V6 必跑）
go test -run TestXxx -v ./...              # 单测
go test -bench=. -benchmem -count=6 .      # 基准（对比 v1.6.0 终态 p6-final.txt）
benchstat docs/plans/p6-final.txt new.txt  # 显著性对比
golangci-lint run                          # 提交前
```

### 1.2 提交规范（Conventional Commits + scope，破坏性带 `!`）
- 每个**子阶段完成**即提交一次，**不要攒到最后**；子阶段改动多时按功能点再拆。
- 前缀：`feat / fix / refactor / perf / docs / test / chore`，带 scope；**破坏性变更**用 `!`（如 `refactor(validators)!: ...`），正文加 `BREAKING CHANGE:` 脚注。
- 示例：
  - `chore(deps)!: bump go to 1.21 and adopt stdlib slices/maps`
  - `refactor(validators)!: unify built-in validators to reflect.Value params`
  - `refactor(datasource)!: TryGet returns reflect.Value carrier; drop source boxing`
  - `feat(register): support registering custom type validators`
  - `chore(mod)!: move module path to github.com/gookit/validate/v2`
- > 注：`validate/` 现已是 git 仓库，当前在 `dev_v2` 分支，正常提交即可（v1.6.0 计划 §1.2 的"无 .git"说明已过时）。

### 1.3 本文档维护
- 每完成一个任务勾选 `[ ]` → `[x]`。
- 每完成一个阶段，在「验收门禁」回填实测数据 / 结论；破坏性变更同步补进 §5。
- 偏离设计/计划时，在该阶段就地记录"实际实现 + 偏离原因"（沿用 v1.6.0 计划的对账风格）。

### 1.4 文件行数约束（AGENTS.md：单文件 ≤ 1000 行）
- 当前 `validators.go`(1305) 已超标——V3 改造它时**顺带按验证器类别拆分**（比较类 / 字符串类 / 长度数量类 / 网络格式类…）回落 ≤1000，是本次合理拆分时机。
- 新增代码避免继续堆进 `validating.go`/`validators.go`。

### 1.5 改动量确认（AGENTS.md）
- V3/V4 必然 >3 文件、>100 行业务代码——按 AGENTS.md，**每阶段开工前向用户确认范围**，再按子阶段小步提交。

## 2. 阶段依赖与顺序

```
V0 护栏 ─► V1 Go1.21 ─► V2 清deprecated ─► V3 验证器reflect.Value ─► V4 DataFace+消装箱 ─► V5 自定义类型 ─► V6 Factory/Pool ─► V7 /v2路径 ─► V8 收尾
 (低)      (低)         (低)              (高)                      (高)                  (中)            (低)            (中机械)      (低)
```

- **基础先行**：V1（Go 版本）、V2（清理）低风险、可独立合并，先做以净化地基。
- **核心破坏**：V3 → V4 顺序（验证器先具备消费 `reflect.Value` 的能力，再由 `DataFace` 源头直供，完成往返消除的最后一跳）。两者均可独立编译通过、独立提交；**顺序可调换**，但 V4 的"源头不装箱"收益需 V3 落地后才完整。
- **机械收尾**：V7（`/v2` 路径 sweep）放靠后，一次性改全部 import，避免开发期 churn。
- 每阶段以"全量测试绿（golden 同步更新后绿）+ 基准不回退"为进入下一阶段前提。

### V3 — 验证器统一 `reflect.Value` 入参 + 删 `convertArgsType` 🔴高风险

**目标**：内置验证器内部统一收 `reflect.Value`；管线对内置验证器直传 `fieldValue.rv` + 构建期已转换的参数，删除运行期 `convertArgsType`（validating.go:514）整套按调用类型转换。这是 v2.0 的**头号破坏性变更**，也是 v1.6.0 P3「args 保持字符串、运行期转换」妥协的最终解。

**关键设计（见 CHANGELOG V2-TODO + 设计 §4.5 / §6.0 协同点）**
```go
// 公共签名统一为 any（破坏点：原 Gt(val any, dstVal int64) → Gt(val, dstVal any)）
func Gt(val, dstVal any) bool { return gt(reflect.ValueOf(val), reflect.ValueOf(dstVal)) }

// 内部实现统一收 reflect.Value（管线直传 fieldValue.rv，零额外 ValueOf）
func gt(val, dstVal reflect.Value) bool { /* ... */ }
```
- 管线（`valueValidate`/`callValidator`）对内置验证器：用 `fieldValue.rv` + 模板里**构建期已转换**的 args 直调内部 `reflect.Value` 版本，**删除 `convertArgsType`**（validating.go:304 调用点 + :514 定义）。
- 落地 v1.6.0 P3 搁置的「构建期预转换 args」：`RuleTpl.Args` 在类型构建期按验证器签名转换好且不可变（设计 §4.2）。

**任务**
- [ ] 设计内部 `reflect.Value` 版验证器的统一形态（命名约定：导出 `Xxx(any...)` 薄封装 → 内部小写 `xxx(reflect.Value...)`）
- [ ] 按类别分批改造 `validators.go` 的 66 个导出验证器（每类一提交）：
  - [ ] 比较类（`Gt/Lt/Gte/Lte/Eq/Ne/Min/Max/Between`…）
  - [ ] 长度/数量类（`Len/MinLen/MaxLen/Size`…）
  - [ ] 字符串/格式类（`IsEmail/IsURL/IsIP`…，多已是 `any`/`string`，按需统一）
  - [ ] 集合类（`In/NotIn/Enum`…）
- [ ] 管线删除 `convertArgsType`：内置验证器走 `reflect.Value` 直调；args 改构建期转换并存入不可变 `RuleTpl.Args`
- [ ] **拆分 `validators.go`**（1305 行）按类别为多文件，回落 ≤1000 行
- [ ] **同步更新 golden**：`regression_compat_test.go` 的 `%#v` 期望随 args 类型语义变化更新；提交说明标注"非回归，签名变更所致"
- [ ] 新增/调整验证器单测覆盖新签名

**坑（重点）**
- **golden 会变**：这是唯一允许 golden 变更的阶段——必须人工逐条核对每个变更是签名所致而非逻辑回归，并在提交里说明。
- 自定义验证器（用户注册的 func）签名不受此约束——内部仍走反射调用，复用 `fieldValue.rv`（v1.6.0 P1 已铺路）。
- 构建期 args 转换需保留 nil 语义、保留 `*T(nil)` 与 `T` 零值的区分（v1.6.0 P3 的克隆/nil 规则延续）。
- `convertArgsType` 删除后，所有依赖其运行期 in-place 改写 args 的隐式行为须确认已被构建期转换覆盖。

**验收门禁**：golden 更新后 `count=3` 绿；全量 + race 绿；`convertArgsType` 相关分配清零，StructFlat allocs 较 v1.6.0 终态进一步下降；`validators.go` 拆分后各文件 ≤1000 行。
**提交**：`refactor(validators)!: unify built-in validators to reflect.Value params`（+ 按类别多次提交 + `perf(exec): drop convertArgsType, dispatch reflect.Value directly`）

### V4 — `DataFace` 扩展 + 消除源头 `Interface()` 装箱 🔴高风险

**目标**：`DataFace.TryGet` 返回携带 `reflect.Value` 的载体，消除 `data_source.go` 缓存命中后 `fv.Interface()` 拆箱（痛点 A 第 1 跳）；`GetWithDefault` 返回 `*fieldValue` 全管线流转，落地设计 §4.3 **完整版**。同时把 `BindStruct/SafeVal` 纳入 `DataFace` 接口。

**关键设计（见设计 §4.3 / §0.1 / §6.0）**
```go
// 现状 DataFace（data_source.go:57）
type DataFace interface {
    Type() uint8
    Src() any
    Get(key string) (val any, exist bool)
    TryGet(key string) (val any, exist, zero bool)   // ← 破坏点：改返回 reflect.Value 载体
    Set(field string, val any) (any, error)
    Create(err ...error) *Validation
    Validation(err ...error) *Validation
    // v2.0 新增（破坏点：自定义实现者须补齐）
    // BindStruct(ptr any) error
    // SafeVal(key string) (any, bool)
}
```
- `TryGet` 返回值携带 `reflect.Value`（如 `(*fieldValue, bool)` 或 `(reflect.Value, exist, zero)`），`StructData` 取字段时直接塞入已有的 `reflect.Value`，**不再 `Interface()` 拆箱**。
- `GetWithDefault` 改返回 `*fieldValue`（内部），公共 `Get/Safe` 签名保持（内部取 `.src`）。
- 三个数据源 `MapData`/`FormData`/`StructData` 适配新接口；map/form 源无源 `reflect.Value`，载体懒构造（与现状等价）。

**任务**
- [ ] 定型 `DataFace.TryGet` 新返回形态（载体类型 + nil/zero 语义）
- [ ] 三数据源实现新 `TryGet`：`StructData` 直供 `reflect.Value`；`MapData`/`FormData` 懒构造载体
- [ ] `BindStruct`/`SafeVal` 提升进 `DataFace` 接口，三源补齐实现
- [ ] `GetWithDefault` 返回 `*fieldValue`，贯穿管线（消除痛点 A 第 1、2 跳）
- [ ] 更新所有 `DataFace` 调用点 + 内部桥接（v1.6.0 P1 的 `vfv=nil` 回退点复核）
- [ ] race 测试 + 内存对比

**坑**
- 自定义 `DataFace` 实现者是受影响群体——§5 清单 + 迁移指南须给出"如何适配新接口"。
- typed-nil vs untyped-nil 语义（v1.6.0 P1 的 `nilRVal`/`Invalid` 处理）在新载体上须等价复刻。
- `Set` 路径 v1.6.0 保守未改（仍 `FieldByName` + nil 指针初始化）——本阶段视情况一并梳理或继续保守，按"有不确定就不动"。

**验收门禁**：golden `count=3` 绿；全量 + race 绿；StructFlat / Nested 较 v1.6.0 终态 allocs 再降（源头装箱消除可量化）。
**提交**：`refactor(datasource)!: TryGet returns reflect.Value carrier; promote BindStruct/SafeVal to interface`

### V6 — opt-in Factory + sync.Pool（Struct/Map）🟢低风险

**目标**：落地 v1.6.0 延后的可选 `Factory` + pooled `Validation` + `Release()`（CHANGELOG V2-TODO 第 3 项；设计 §4.7）。`Val/Var` 的池化 v1.6.0 已做（`fix(value)` 67292f0），本阶段补 `Struct/Map` 工厂。

**关键设计（见设计 §4.7）**
```go
f := validate.NewFactory()
v := f.Struct(&user)   // 内部 pool 取 Validation + 命中 typeCache
defer v.Release()      // 归还前 Reset 全字段
v.Validate()
```

**任务**
- [ ] `Factory` + `pool *sync.Pool`；`Release()` 前 `Reset` 全字段（防串扰）
- [ ] 默认 `validate.Struct/Map` 行为、生命周期**完全不变**（opt-in）
- [ ] `-race` + 内存泄漏 / 数据串扰测试（参照 v1.6.0 `TestVal_concurrent` 风格，多 goroutine 混合路径）

**坑**
- `Reset` 不彻底是池化头号风险——逐字段核对，`-race` 复现验证。
- opt-in、非默认：默认基准须零回退。

**验收门禁**：default 基准零回退；Factory heavy-reuse 场景 allocs 显著下降；race + 串扰测试绿。

## 6. 风险与回滚

| 风险 | 阶段 | 缓解 / 回滚 |
|---|---|---|
| 验证器签名改造遗漏某验证器 / golden 误判 | V3 | 按类别小步提交；golden 逐条人工核对（签名变更 vs 逻辑回归）；不一致即回滚该类提交 |
| 删 `convertArgsType` 后丢失某隐式运行期转换行为 | V3 | 构建期转换须覆盖原 in-place 改写的全部分支；golden + 全量回归守护 |
| `TryGet` 返回值改型遗漏 typed-nil / zero 语义 | V4 | 复刻 v1.6.0 P1 的 `nilRVal`/`Invalid` 处理；专项用例锁定 |
| 自定义 `DataFace` 实现者无声破坏 | V4 | §5 清单 + 迁移指南显式说明；CHANGELOG 醒目标注 |
| V5 自定义类型 API 设计不当致返工 | V5 | 开工前先写一页小设计评审；并发安全参照 `typeCache` |
| 池化 `Reset` 不彻底串数据 | V6 | opt-in 非默认；`Release` 前 Reset 全字段；`-race` 复现 |
| `/v2` sweep 漏改 import | V7 | 放靠后一次性 sweep；`go build ./...` 兜底；examples 独立 go.mod 单独处理 |

**回滚策略**：每阶段独立提交、独立可编译；出问题回滚到上一阶段末尾绿色提交。V1/V2/V6 可独立合并；V3/V4 是核心破坏对，建议成对验证后再继续。即使只完成到 V4，已具备 v2.0 的核心 API 精简价值。
