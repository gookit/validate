## 0. 总览

把 `docs/validate-v2-design.md` §6.1 的 P0–P6 落地为 v1.6.0。核心三件事：
1. **TypeMeta**：缓存 struct 类型级反射元数据（字段索引、tag、预解析规则模板）。
2. **FieldValue**：贯穿管线、携带 `reflect.Value` 的值载体，消除 `reflect.Value→Interface()→reflect.ValueOf` 往返。
3. **管线拆分 + 实现下沉 internal**，提升可维护性。

**红线（每阶段都必须满足）**：
- 现有全部测试零回归（`issues_test.go`/`issues_x2_test.go`/`validation_test.go`/`validators_test.go`/`data_source_test.go`/`morecase_test.go` 等）。
- 公共 API（包级函数、`Validation` 公共方法、`DataFace` 接口、内置验证器函数签名、struct tag 语义）**不变**。
- 基准不回退（以 P0 基线为准）。

> ⚠️ 破坏性内容（验证器统一 `reflect.Value` 入参、`DataFace` 扩展、注册自定义类型、`/v2`、升 Go 1.21）**不属于本计划**，留待 1.6.0 稳定后的 v2.0.0。见设计文档 §6.0。

### P1 — FieldValue 值载体 🟢低风险

**目标**：引入携带 `reflect.Value` 的值载体，先在热点函数内部替换重复 `reflect.ValueOf`，**对外行为完全不变**。

**关键设计（见设计 §4.3）**
```go
// 新文件：fieldvalue.go（根包；P4 再考虑迁 internal）
type FieldValue struct {
    name string
    src  any
    rv   reflect.Value  // 懒：reflect.ValueOf(src)；若来源已有 reflect.Value 则直接塞入
    rt   reflect.Type   // 懒
    real reflect.Value  // 懒：reflect.Indirect(rv)
    rvInit, realInit bool
    zero, empty int8    // 0未知 1是 2否（懒）
}
func (f *FieldValue) RV() reflect.Value   // 懒构造并缓存
func (f *FieldValue) Real() reflect.Value // 去指针，统一指针语义
func (f *FieldValue) IsZero() bool        // 懒
func (f *FieldValue) IsEmpty() bool       // 懒，复用 RV()，替代 IsEmpty(any) 的再反射
```

**任务**
- [x] 新建 `fieldvalue.go`：不导出结构 `fieldValue` + `rV/rT/realV/isZero/isEmpty` 方法（懒求值、三态缓存）
  - 决策：**用不导出名 `fieldValue`**（不过早锁定公共 API；导出留待后续）
- [x] 提供两种构造：`newFieldValue(name, src)`（map/form 用）/ `newFieldValueRV(name, rv)`（struct 用，不调 `.Interface()` 还原；**P2 才接入**，P1 先定义补全 API）
- [x] 新建 `fieldvalue_test.go`：覆盖普通值/空串/`0`·`false`/nil·空 slice·map/`*T` 有效/`*T(nil)`/`any(nil)`/`**int`；`isEmpty` 对 18 输入逐一断言 `== IsEmpty(src)`（40+ 子用例全过）
- [x] 改造 `callValidatorValue`（validating.go:542）：新增末位参数 `vfv *fieldValue`，非 nil 时复用 `vfv.realV()` 替代第二次 `reflect.ValueOf(val)` 与指针解引用
- [x] 改造 `valueValidate`（validating.go:300）：复用 `fv.rV()`，并把载体经 `callValidator` 贯穿到 `callValidatorValue`
- [x] 指针/零值语义集中到 `fieldValue.realV/isEmpty`（保留公共函数不动）

**注意 / 坑（已落实）**
- `IsEmpty(any)`（validators.go:542）签名不动；`fieldValue.isEmpty()` 复用 `ValueIsEmpty`（util.go:185），三路径与 `IsEmpty(any)` 对齐。
- typed-nil vs untyped-nil：`rV()` 对 `src==any(nil)` 返回 `nilRVal`；`valueValidate` 中 `src==nil` 时强制 `valKind=Invalid` 还原原行为（`rftVal` 仅 Slice 分支消费，nil 永不进入）。
- **行为等价铁律**：任何"值与载体不一致"的点（类型转换后、slice 子元素、nil）一律传 `vfv=nil` 回退原 reflect 逻辑。监督者逐行核对 diff，确认 100% 等价。

**验收门禁**：✅ 全量测试 + `-race` 全绿；基准全部持平或微降（StructFlat ~-2.3%），**无回退**。
- ⚠️ **门禁修正（真实发现）**：原预期"StructFlat allocs 下降"**不成立且非回退**——P0 剖析已知 allocs 集中在解析阶段；且所有基准 struct 用**内置**验证器，命中 `callValidator` switch 快路径，**根本不走 `callValidatorValue` 反射路径**，故 P1 的反射去重对这些基准无 alloc 影响。P1 的反射去重仅惠及**自定义验证器**路径（正确性由海量回归用例保证），其 alloc 收益需自定义验证器基准才能体现（可于 P4 补）。P1 的核心交付是**抽象层落地 + 为 P2 预留 `newFieldValueRV`**。
**提交**：`perf(fieldval): introduce fieldValue carrier to dedup reflect in validate path`

### P2 — 类型元数据缓存 TypeMeta 🟡中风险

**目标**：同一 struct 类型的字段索引 / kind / tag 只构建一次并缓存；字段访问由 `FieldByName`（O(n)）改 `FieldByIndex`（O(1)）。

**关键设计（见设计 §4.2 / §4.6）**
```go
// cache.go（替换现有全注释内容）
type FieldMeta struct {
    Index   []int        // FieldByIndex 路径
    Name, Path string
    Kind   reflect.Kind  // 去指针后
    IsPtr  bool
    Elem   elemClass     // leaf / structKind / sliceOfStruct / mapOfStruct...
    ValidateRule, FilterRule, OutputName, Label string
    Messages map[string]string
}
type TypeMeta struct {
    Type   reflect.Type
    Fields []*FieldMeta
    byName map[string]*FieldMeta
    // P3 再填充：StaticRules / DynamicFields
    Labels, Outputs map[string]string
    ImplConfig, ImplTranslates, ImplMessages bool
}
type typeKey struct { rt reflect.Type; tagVer uint32 }
var typeCache sync.Map  // map[typeKey]*TypeMeta
func getTypeMeta(rt reflect.Type) *TypeMeta  // 命中即返回；未命中构建后存入
```

**任务**
- [x] `cache.go`(228行)：`fieldMeta`/`typeMeta`/`typeKey` + `getTypeMeta`(Load→build→LoadOrStore) + `buildTypeMeta`（**纯类型遍历 `t.Field(i)`，不碰任何值**；递归 struct-of-struct，跳过 `timeType`/非导出；slice·map-of-struct 仅标记入 `dynamicFields` 不展开；5 个 tag 一次性 `Tag.Get` 存 `fieldMeta` 供 P3 用）
- [x] tag 配置版本：`Config()`/`ResetOption()` 末尾 `atomic.AddUint32(&tagVer,1)`；`getTypeMeta` 用 `typeKey{rt,tagVer}` → 配置变更旧缓存自然不可达
- [x] 公共 `ResetTypeCache()`（`typeCache.Range`+`Delete`，并发安全）
- [x] `StructData` 加 `meta *typeMeta`；`FromStruct` `data.meta = getTypeMeta(typ)`
- [x] `TryGet` **仅顶层导出字段**经 `topFieldMeta` 命中后用 `FieldByIndex`，否则回退 `FieldByName`；嵌套/slice/map 路径不动
- [x] `Create` 三处 `Implements` 改读 `d.implFaces()`（meta 缺失回退 live 检查，行为等价）
- [x] `cache_test.go`：命中同实例 / `*T`·`T` 共享 / 嵌套 Index / 非导出(默认跳过·开启后含) / Implements / tagVer 失效 / ResetTypeCache / 50 并发 `getTypeMeta`(`-race`) **+ 递归类型回归(见下)**

**坑（已落实）**
- 非导出/匿名字段：`topFieldMeta` 仅对"meta 命中 + 单级 Index + 首字母大写"用 `FieldByIndex`，其余回退 `FieldByName`（规避非导出 panic）。
- **🐞 监督者修复的真实回归（agent 漏掉）**：`buildTypeMeta` 走**类型树**，对自引用/互递归 struct 类型（`type Node struct{Next *Node}`）**无限递归→栈溢出**（原 `parseRulesFromTag` 走值树，nil 指针天然终止，故原来正常）。**修复**：`walk` 加**祖先链环检测**（同类型在兄弟分支仍可展开，只切断祖先环；环上 struct 字段记入 `dynamicFields` 交 P3 按值处理）。已加 `TestGetTypeMeta_cyclicType`（self-ref / mutual-rec / 端到端不挂起）永久锁定。
- `Set` 路径**保守未改**（仍 `FieldByName`，含 nil 指针初始化，按"有不确定就不动"）。
- `timeType` 已在 build 递归中排除。

**验收门禁**：✅ 全量测试 + `-race` 全绿（含递归类型回归）。

### P3 — 规则模板缓存 RuleTpl + build/instantiate 两段式 🟡中风险

**目标**：把规则收集拆成"类型级构建（缓存）"+"值级实例化"，并把规则字符串解析、别名解析、参数类型转换前移到构建期（最大增益点之一）。

**关键设计（见设计 §4.4）**
```go
type RuleTpl struct {
    Path, Validator, RealName string
    NotRequired bool
    Args        []any      // 构建期已按验证器签名转换好，运行时不可变
    FuncMeta    *funcMeta  // 全局/内置验证器构建期绑定；struct 方法验证器留空、运行时解析
    FilterRule  string
}
// TypeMeta 补充：
//   StaticRules   []*RuleTpl   // 路径恒定（leaf、struct-of-struct）
//   DynamicFields []*FieldMeta // slice/map-of-struct，需按值展开
```
拆分 `parseRulesFromTag`（data_source.go:270）：
- `build(rt)`：只遍历**类型**，解析全部 tag → 生成 `StaticRules` + 翻译表；slice/map-of-struct 记入 `DynamicFields`（递归缓存元素类型 meta）。
- `instantiate(value, v)`：挂载 `StaticRules`（零反射）；仅对 `DynamicFields` 按真实值展开元素路径（`items.0.name`...）。

> **实现决策（比原设计更安全）**：放弃"构建期预转换 args + 全量静态展开"（`%#v` 黄金测试无法验证 arg 类型转换且涉及自定义验证器/nil 语义风险）。改为 **静态快路径 + 动态回退**：
> - **静态类型**（字段图沿非指针 struct 递归无 ptr-to-struct / slice·map-of-struct）：规则集与值无关 → 用**零值实例跑一次现有 `parseRulesFromTag` 生成模板快照**并缓存（复用同一解析代码 ⇒ 保证逐字节一致），实例化时克隆模板（args 逐实例拷贝、保留 nil 语义、trans replay）。
> - **动态类型**（ptr/slice/map-of-struct、递归类型）：**完全走原 `parseRulesFromTag` 不变**，零风险。
> - args 仍保持收集期形态（字符串），运行时 `convertArgsType` 不变（"构建期预转换 + 共享不可变 rule"留待 P5/v2，因 newEmpty 才是残余分配大头）。

**任务（已完成）**
- [x] P3a：`regression_compat_test.go` 黄金安全网（27 用例，`%#v` 含 arg 类型；钉死 enum/notIn 合并、regexp 不拆、nil-vs-empty args）—— **已独立提交 `48e2746`**
- [x] `struct_rules.go`(501行)：`computeIsStatic` 分类 + `buildRuleTemplate`(零值快照) + `staticTemplate`(sync.Once) + `instantiateStatic` + `cloneRule`/`cloneFilterRule`
- [x] `typeMeta` 加 `isStatic`/`tplOnce`/`tpl`；`buildTypeMeta` 末尾算 `isStatic`
- [x] `Create` 分发：`if d.meta.isStatic { instantiateStatic } else { parseRulesFromTag }`
- [x] 把 `parseRulesFromTag`、`loadMessagesFromTag` **逐字节移动**到 `struct_rules.go`（git 提取对比零 diff）→ `data_source.go` 1078→**838 行（≤1000）**
- [x] `cache_test.go` 加 `TestComputeIsStatic` + `TestStaticTemplate_concurrent/isolation`
- [x] 临时对照：20 个 static fixture 强制 static 路径 dump == dynamic 路径 dump 逐字节相等（已删临时测试）

**注意 / 坑（已落实）**
- **args 逐实例拷贝**：验证期 `convertArgsType` 原地改写 `r.arguments`，共享模板会污染/竞态 → `cloneRule` 每实例新建 args 切片；**保留 nil 语义**（模板 nil args 克隆为 nil，非空切片）。
- 模板懒构建 `sync.Once` 并发安全；模板只读不可变。
- 自定义/struct 方法验证器：static 模板由现有 `parseRulesFromTag` 生成，行为与原一致；含此类验证器的结构仍能正确静态化（解析逻辑不变）。
- 自引用/递归类型必含 ptr-to-struct → `computeIsStatic` 判 dynamic，不会进静态模板。

**验收门禁**：✅ **黄金测试 `count=3` 逐字节绿（硬门禁达成）**；全量 + `-race` 全绿；`data_source.go` 838 行。

### P4 — 验证管线拆分 + 下沉 internal 🟡中风险

**目标**：拆分巨函数提升可维护性（痛点 D）；把纯内部机制迁入 `internal/`（痛点 E）。**行为零回归**。
