# #324 嵌套表单绑定（multipart/form-data）— 设计方案（不改码，待评审）

> **目的**：为 #324「multipart 嵌套字段不绑定到嵌套 struct」定设计，评估改动面/风险/范围后再实施。
> **状态**：✅ **已按本设计实施** — Part1 `bbdad99`、Part2 `a9c3fd3`。决策落地：仅对象嵌套；归一统一在 `FromURLValues`（§6.2）；默认自动、无 opt-in 开关（§5.A）。**数组 bracket：探针证伪"复用 maputil"假设后决定暂不做（§4.1）。**
> **日期**：2026-06-05
> **前置**：复现已落 `TestIssue_324_v2`（bracket `address[street]` 与 dot `address.street` 都不绑定，Address 子字段为空）。

---

## 1. 根因（已定位）

1. **取值平铺**：`FromURLValues`（`validate.go:444`）把表单键**原样**塞进 `d.Form`(url.Values)。客户端发 `address[street]=Main St`，`d.Form` 的键就是字面量 `"address[street]"`。
2. **绑定不展开**：`BindSafeData`（`validation.go:577`）把 `v.safeData`（**扁平** `map[string]any`）`Marshal` 成 JSON 再 `Unmarshal` 进 struct。扁平键 `"address[street]"` / `"address.street"` 在 JSON 里只是**顶层扁平键**，不会变成嵌套对象，所以 `Address.Street` 始终空。
3. **校验取不到**：即便用户加规则 `v.StringRule("address.street", ...)`，`FormData.Get("address.street")` 查 `d.Form["address.street"]` 也命不中（实际键是 `address[street]`），nested 字段连校验都进不去 → 不进 safeData → 更不会绑定。

→ 三处叠加：bracket 键不归一、校验取不到、绑定不展开。

---

## 2. 设计（两部分 + 一个前提）

### 2.1 Part 1 — bracket → dot 归一（取值入口）

在表单摄入时把 bracket 键归一为库内统一的**点路径**：

- `address[street]` → `address.street`
- `address[street][no]` → `address.street.no`（多层对象）
- （数组见 §4）`items[0][name]` → `items[0].name`（保留数字 `[N]`，仅非数字 `[k]`→`.k`）

落点：`FromURLValues`（`validate.go:444`）加一个 `normalizeFormKey(key)`，`data.Add(normalizeFormKey(key), val)`。归一只影响**含 `[` 的键**，普通键零改动。

归一后整条链路统一用点路径：`d.Form["address.street"]` 可被 `FormData.Get`、规则、`tryGet` 正常命中。

### 2.2 Part 2 — 绑定时把扁平点路径展开成嵌套 map（BindSafeData）

`BindSafeData` 现在 `Marshal(v.safeData)`。改为：先用 `maputil.SetByPath` 把扁平点路径 safeData 展开成**嵌套 map**，再 Marshal+Unmarshal：

```go
nested := make(map[string]any, len(v.safeData))
for k, val := range v.safeData {
    _ = maputil.SetByPath(&nested, k, val) // "address.street" -> {address:{street:val}}
}
bts, err := Marshal(nested)
// Unmarshal(bts, ptr) 不变
```

- 普通键 `name` → `SetByPath` 等价于顶层 set（行为不变）。
- 点路径键 `address.street` → 嵌套 `{address:{street:...}}` → 绑定到 `Address.Street`。✓
- `maputil.SetByKeys`（`goutil/maputil/setval.go:28`）已支持 `key[N]` bracket 形式建 slice，故数组路径（§4）天然可走同一展开。

### 2.3 前提 — nested 字段必须被校验才进 safeData

`BindSafeData` 只绑定 safeData（"安全数据"=校验过的字段）。所以 nested 字段要绑定，必须有规则让它进 safeData：

```go
data := validate.FromURLValues(form) // form: address[street]/address[city]/name
v := data.Create()
v.StringRule("name", "required")
v.StringRule("address.street", "required") // 用户声明 nested 规则(点路径)
v.StringRule("address.city", "required")
v.Validate()
v.BindStruct(&member) // Part1+Part2 后, 嵌套正确绑定
```

这与 map/form 源的既有语义一致（声明要校验/收集的字段）。#324 的 Goravel 场景由框架按目标 struct tag 生成这些规则，对应到本库即"加点路径规则"。**本设计不改 safeData 语义**（不做"绑定全部源数据"），保持最小侵入。

> 备选（更大、不在本期）：提供 `BindRequest`/`BindAll` 这类"绑定全部源数据(含未校验)"的方法。这是另一个特性，单列。

## 3. 回归风险与边界

| 风险点 | 评估 | 处置 |
|---|---|---|
| Part1 改 `d.Form` 键 | 仅影响**含 `[` 的键**；现有用例无人用字面 bracket 键 | 跑全量；`normalizeFormKey` 对无 `[` 键原样返回 |
| Part1 也作用于 query（`FromQuery` 复用 `FromURLValues`） | query 用 bracket 同属表单惯例，归一一致合理 | 文档说明；如不想影响 query 可只在 multipart/form 分支归一（见 §6 决策） |
| Part2 展开**所有**点路径键 | 仅含 `.` 的键受影响；未知嵌套键被 `json.Unmarshal` 安全忽略 | flat 键(`name`)行为不变；现有 bind 测试全用 flat 键，无回归 |
| safeData 里 `ports.*.container_start`（#262 通配键） | 展开成 `{ports:{"*":{...}}}`，struct 无 `*` 字段 → 被忽略 | 无害（本就不绑定）；可选：跳过含 `*` 的键 |
| safeData 里 `Samples.0.Val`（struct 级联，bare 数字） | 展开成 `{Samples:{"0":{Val}}}`(map)，绑 slice 失败 | 现状扁平键**本就**绑不进 slice，结果不变；struct 源极少用 BindStruct |
| golden `TestRuleCompat` | 不碰规则收集/校验执行，只碰取值键与绑定 | 必跑 ×3 逐字节确认 |

**净判断**：Part2 只动"含点键"，flat 主路径零变化；Part1 只动"含 `[` 键"。回归面窄，靠 golden+全量+race 兜底。

## 4. 数组下标 bracket（`items[0][name]`）— 是否本期做

- maputil `SetByKeys` 已支持 `key[N]` 建 slice（`setval.go:76` parseArrKeyIndex）。
- 归一规则需区分：数字 `[0]` **保留** bracket（供 slice），非数字 `[street]`→`.street`（对象）。
- 校验侧：slice 元素规则可用 `items.*.name` 通配（已有 `.*` 机制）或显式 `items.0.name`。
- 代价：slice 索引边界、空洞、与 `.*` 校验的衔接、JSON marshal 的 `[N]`→数组形态需逐一测。

**建议**：**Part 1 期只做对象嵌套**（#324 原型即对象嵌套）；数组 bracket 留作 Part 2 单独迭代，避免一次摊大。归一函数预留"数字段保留 bracket"的钩子即可。

### 4.1 后续评估结论（2026-06-05，探针验证）— ❌ 暂不做

实施对象嵌套后对数组 bracket 做了探针验证，**推翻了"复用 maputil `[N]`"的简化假设**，暴露两个独立拦路问题，决定**暂缓不做**：

- **绑定：maputil 建不出 slice**。`maputil.SetByPath("items[0].name", v)` 产出字面量 map 键 `{"items[0]":{...}}`（非切片）；`SetByPath("items.0.name", v)` 产出 `{"items":{"0":{...}}}`（map，键 `"0"`），JSON 是对象非数组，**绑不进 `[]Item`**。→ 需**自研"数字段→slice 索引"的嵌套构建器**（不可复用 maputil），并生成真正的 JSON 数组。
- **校验：表单数组无法枚举**。静态显式索引规则 `items.0.name` 能命中、进 safeData（但要求预知元素个数）；通配 `items.*.name` **失败**（表单是扁平 `url.Values`，`.*` 不枚举扁平的 `items.N.name` 键）。→ 动态长度数组连"让元素进 safeData"都做不到，需**改造表单 `.*` 枚举逻辑**。

**结论**：数组 bracket 实为两个非平凡子系统（自研 slice 嵌套构建器 + 表单数组枚举），远超"复用 maputil 的小扩展"，且动态长度才是真需求。性价比低，**暂不实现**；对象嵌套已覆盖 #324 主场景。若将来要做，须按上述两点单独立项设计。

## 5. 开关门控（需决策）

bracket 归一是否默认开启？

- **A. 默认开启（自动）**：最贴近 issue「就该自动绑」的期望；blast 半径仅"含 `[` 的键"，风险可控。**推荐**。
- **B. 选项开启（opt-in）**：加 `GlobalOption.ParseFormBrackets bool`（默认 false），用户显式开启。最稳但不"开箱即用"，且又多一个全局开关。

倾向 **A**：含 bracket 的表单键在现网几乎只可能是嵌套意图，自动归一符合直觉；如担心，可在 §6 决策为 B。
