## 1. #262 — filter 无法对切片元素逐个应用

### 1.1 根因（已定位）

- `FilterRule.Apply`（`filtering.go:142`）对每个字段 `v.tryGet("ports.*.container_start")`。
- Map/JSON 源走 `MapData.TryGet → maputil.GetByPath`，通配 `.*` **收集成整切片** `[]any{80}`（`data_source.go:113`）。
- 随后把 filter `int` 应用到**整个 `[]any{80}`**：`filter.Apply("int", []any{80})` → 把 slice 当单值转 int → `convert value type error`。
- 验证器侧没有这个问题：`valueValidate`（`validating.go:286`）检测到 `dotStarNum>0 && valKind==Slice` 会走 `validateWildcardSlice`（`:391`）**逐元素**校验。filter 链路缺这一段展开。
- 显式索引 `ports.0.container_start` 取到标量 `80`，filter 正常 → 这就是当前 workaround。

### 1.2 关键衔接点（为何 filter 修好后验证器能直接复用）

`tryGet`（`validation.go:499`）**先查 `v.filteredData[key]`**（`:505`）再查源数据。filter 把结果写进 `v.filteredData["ports.*.container_start"]`（`filtering.go:190`），验证器对同一通配 key 取值时**优先命中 filteredData**。

→ 只要 filter 把「逐元素过滤后的**收集切片** `[]any{80(int)}`」写回 `filteredData[通配key]`，验证器的 `validateWildcardSlice` 读到的就是它期望的切片形态，逐元素校验自然跑通。**存储形态天然对齐，无需改验证器。**

### 1.3 设计

改 `FilterRule.Apply` 单个函数：

```
for each field in r.Fields():
    val, exist, zero = v.tryGet(field)
    ... // 默认值分支不变

    if strings.Contains(field, ".*") && val 是 slice:
        // 逐元素应用 filter 链，收集成新切片
        out = make([]any, len)
        for i, elem := range val:
            ev = elem
            for each (name, args) in filters:
                ev, err = apply(name, ev, args)   // 复用现有 filter.Apply / callCustomFilter
            out[i] = ev
        newVal = out
    else:
        ... // 现有标量逻辑不变

    v.filteredData[field] = newVal   // 通配 key 存收集切片；标量 key 存标量
```

### 1.4 范围与边界（需拍板）

1. **写回源数据**：`updateValue`（`validation.go:598`）对 **Map/Form 是 no-op**（仅 struct 源 `Set`）。
   - Map/JSON（**本 issue 场景**）：靠 `filteredData` 承载，**无需写回源**，最简单。✅ 优先支持。
   - struct 源逐元素写回需要按 `ports.0.x`/`ports.1.x` 索引 Set，`updateValue` 现在会 `TrimSuffix(field,".*")` 再整体 Set，**不支持逐元素**。→ 建议**本期不动 struct 源的通配 filter 写回**，或单列子任务。
2. **多级通配** `a.*.b.*.c`：`validateWildcardSlice` 用 `flatSlice` 拍平多层。filter 本期可**只支持单级 `.*`**（issue 即单级），多级标注为后续扩展，避免一次摊大。
3. **空切片 / 元素缺字段**：保持与现有 `!exist||zero→默认值` 一致；空切片 `[]any{}` 走默认值分支或原样。

### 1.5 测试改动

- `TestIssue_262_v2`「wildcard filter ...still-broken」→ 翻成 fixed：`Validate()` 通过，`SafeData()["ports.*.container_start"]` 为 `[]any{80}`（int）或逐元素 80。
- 保留「explicit index workaround」用例。
- 补一个：自定义 filter（`callCustomFilter` 路径）的通配逐元素用例。

### 1.6 影响面 / 风险

- 改动：**1 文件 `filtering.go`，~30–40 行**。
- 风险：**低**（自成一体、参照 `validateWildcardSlice`、衔接点已验证）。
- 必跑：golden `TestRuleCompat ×3` + 全量 + race。golden 无通配 filter 用例，预期不变。

## 2. #283 — 场景（scene）无法用通配/无索引命中切片元素字段

### 2.1 根因（已定位）

- 级联（`struct_rules.go:433` `arrayName=fmt.Sprintf("%s.%d",name,j)`）为切片元素生成**带数字索引**的规则名 `Tags.0.Id`。
- 场景字段白名单在 `isNotNeedToCheck`（`validation.go:700`）里**按字符串精确/前缀匹配**：
  - 用户写 `Tags.*.Id` → `sceneFields={"Tags.*.Id":1}`，对 field `Tags.0.Id` 各前缀（`Tags`、`Tags.0`、`Tags.0.Id`）都不命中 → 返回 true（跳过，不校验）。
  - 用户写 `Tags.Id`（无索引）→ 同理不命中。
  - 只有显式 `Tags.0.Id` 命中 → 当前唯一可用写法。

### 2.2 重要现状（设计取舍的前提）

`isNotNeedToCheck` 的**前缀循环**已支持「父字段名」级联：若场景含 `Tags`（整个切片），field `Tags.0.Id` 在 i=1 命中前缀 `Tags` → 返回 false（**校验**）。
→ 即「`Tags` 一把全选切片下所有子字段」**今天就能用**。本 issue 要的是**选择性**：只校验 `Tags` 元素的 `Id`，不校验 `Name`/`Date` —— 这才需要 `Tags.*.Id` 这种「带通配的叶子选择」。

### 2.3 设计

入口 `isNotNeedToCheck` + 预计算 `sceneFieldMap`（`validation.go:628`）。支持两种写法：

**(A) 通配 `Tags.*.Id`（推荐，语义无歧义）**
- 规范化：把 field `Tags.0.Id` 的**数字段**替换为 `*` → `Tags.*.Id`，在「通配场景集」里查存在性。
- 实现：`sceneFieldMap` 把含 `.*` 的场景项单独收进 `sceneWildcards`（set）；`isNotNeedToCheck` 在精确/前缀都未命中后，若 field 含数字段，规范化后查 `sceneWildcards`。

**(B) 无索引 `Tags.Id`（issue 原写法，有歧义）**
- 规范化：把 field 的数字段**整段删除** `Tags.0.Id`→`Tags.Id`，查「无索引场景集」。
- **歧义**：与一个真实存在、字面叫 `Tags.Id` 的非切片嵌套字段冲突。
- 建议：**优先只做 (A)**；(B) 作为可选项，文档明确「推荐用 `.*` 显式通配」，(B) 仅作兼容且标注歧义。

### 2.4 性能（热路径，必须处理）

`isNotNeedToCheck` 对**每条规则的每个字段**都调用。直接 split+通配比较会拖慢无场景/无通配的常见路径。缓解：

1. `sceneFieldMap` 一次性分区：`exact`（现有 map）+ `wildcards`（含 `.*` 的项，可能为空）。
2. `isNotNeedToCheck` 廉价门控：仅当 `len(sceneWildcards)>0` **且** field 含数字段（`strings.ContainsFunc` 检测 `.<digit>` 段）时，才做规范化匹配；否则维持现有 exact/prefix 逻辑，零额外开销。
3. 规范化只在确需时跑一次，O(段数)。

### 2.5 测试改动

- `TestIssue_283_v2`「scene 'Tags.*.Id' ...still-broken」→ 翻 fixed：该场景下空 `Id` 报 `Tags.0.Id is required`，且 `Name`/`Date` 不报（验证选择性）。
- 「scene 'Tags.0.Id' workaround」保留（仍应可用）。
- 「scene 'Tags.Id'（无索引）」：依 §2.3 取舍决定翻 fixed 还是保留 still-broken（若只做 A 则保留，并补注释指向 `.*`）。

### 2.6 影响面 / 风险

- 改动：**1 文件 `validation.go`**（`sceneFieldMap` + `isNotNeedToCheck` + 一个结构体字段 `sceneWildcards`），**~30–50 行**。
- 风险：**低–中**（热路径性能需门控；无索引写法有语义歧义）。
- 必跑：golden `TestRuleCompat ×3`（含 `regression_compat_test.go:616` scene 用例，仅普通字段名，预期不变）+ 全量 + race。
