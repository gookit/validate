# #292 规则级"逻辑或" — `rule_one_of` 设计方案

> **✅ 已实施**（phase1：仅无参子校验器）。验证全绿：golden ×3 逐字节 / 全量 / race / vet。
> **目的**：为 #292「希望支持完整逻辑条件（或/非）」定一个**可落地、改动可控**的设计：新增一个组合校验器 `rule_one_of`，"满足列出的任一规则即通过"。
> **状态**：已实施（按本稿落地，5 处接线 + `TestIssue_292_v2`）。
> **日期**：2026-06-05
> **背景**：#292 想验证「字段是 IP **或** CIDR」，但 `validate:"ip|CIDR"` 里 `|` 是规则分隔符（逻辑与，两者都要满足）。评论区 @almas-x 提议新增 `one_of` 这类组合规则、任一满足即通过——本设计采纳该思路。

---

## 1. 命名决策：用 `rule_one_of`，不要 `one_of` / `oneof`

**`oneof` 已被占用**：`register.go:134` 里 `"oneof": "enum"` —— 即 `oneof` 是 `enum`（`in`）的别名，语义是"**值**属于给定**集合**之一"（如 `oneof:a,b,c`）。`one_of` 与它仅一下划线之差，极易混淆。

为避免歧义，组合规则命名 **`rule_one_of`**，明确语义是"满足这些**规则/校验器**之一"（区别于"值属于集合之一"）：

| 名称 | 现状/语义 |
|---|---|
| `in` / `enum` / `oneof` | 已存在：**值**∈ 给定集合（`oneof:a,b,c`） |
| **`rule_one_of`**（本设计） | 新增：**值满足**列出的**校验器**之一（`rule_one_of:ip,cidr`） |

（"非/and-of" 等其它逻辑暂不在本设计范围，见 §6。）

---

## 2. 可行性确认（机制已具备，已查代码）

实现 `rule_one_of` 只需"按名解析校验器 + 在同一个值上逐个调用"，这套机制本库已全有：

- **按名解析别名**：`validators.go:132 ValidatorName(name)`（`ip`→`isIP`、`cidr`→`isCIDR`，别名表 `register.go:131 validatorAliases`）。
- **按名取校验器元信息**：`validation.go:299 validatorMeta(name)`（实例 → 全局 → ctx 内建 → struct 方法）。
- **调用校验器**：`validating.go:499 callValidator(v, fm, field, val, args, addNum, vfv)`。
- **目标子校验器存在**：`IsIP`/`IsCIDR` 等为已注册的自由函数（`register.go:76/85`，`validators_string.go:153`）。
- **列表参数收集已有范式**：`rule.go:201 case "enum", "notIn":` 把逗号串收成**单个 `[]string` 参数**。而**结构体 tag 也走 `StringRule`**（`struct_rules.go:355 v.StringRule(name, vRule)`），所以这一处改动**同时覆盖 tag 与程序化两种写法**。

## 3. 设计

### 3.1 用法

```go
// 结构体 tag
type Host struct {
    Addr string `validate:"required|rule_one_of:ip,cidr"`
}
// 程序化
v.StringRule("addr", "required|rule_one_of:ip,cidr")
```

- **语义**：`val` 满足 `ip` **或** `cidr` 任一即通过；全部不满足才失败。
- 与 `required` 组合实现"必填且满足其一"（`rule_one_of` 本身是普通非必填校验器，见 §5）。

### 3.2 实现落点（5 处，约 50–80 行）

1. **参数收集**（`rule.go:201`）：把 `rule_one_of` 加入 enum 范式
   ```go
   case "enum", "notIn", "rule_one_of":
       v.AddRule(field, validator, parseArgString(list[1]))
   ```
   使 `rule_one_of:ip,cidr` 的 `["ip","cidr"]` 作为**单个列表参数** `args[0]` 传入。tag 与 StringRule 同时生效。

2. **新方法**（`validators.go`，与 `Required` 等同处）
   ```go
   // RuleOneOf 满足列出的任一校验器即通过 (#292)。子项为校验器名(支持别名), 仅无参校验器。
   func (v *Validation) RuleOneOf(field string, val any, rules any) bool {
       names, _ := rules.([]string) // 来自 parseArgString
       for _, name := range names {
           realName := ValidatorName(strings.TrimSpace(name))
           fm := v.validatorMeta(realName)
           if fm == nil {
               // 未知子校验器: 见 §4 决策(建议 build 期 panic, 此处保守 continue)
               continue
           }
           if callValidator(v, fm, field, val, nil, 1, nil) {
               return true
           }
       }
       return false
   }
   ```

3. **ctx 绑定**（`validate.go` `ctxValidatorBuilders`，约 207 行）：
   ```go
   "rule_one_of": func(v *Validation) reflect.Value { return reflect.ValueOf(v.RuleOneOf) },
   ```
   使 `validatorMeta("rule_one_of")` 能懒构建并绑定真实 `v`。

4. **dispatch**（`validating.go` `callValidator` switch，约 525 行 enum 附近）：
   ```go
   case "rule_one_of":
       ok = v.RuleOneOf(field, val, args[0])
   ```

5. **错误消息**（`messages.go:153 builtinMessages`）：
   ```go
   "rule_one_of": "{field} did not satisfy any of: %v",
   ```
   （`%v` 渲染子校验器名列表；中文消息按 i18n 机制另配。）

可选：加常量 `RuleOneOf = "rule_one_of"`（`register.go`）。

### 3.3 closest analog（实现时对照）

`rule_one_of` = **enum 的"列表单参数 + 非必填"接线** + **required-family 的"方法绑定 v"接线** 的组合：
- 像 `enum`：1 个列表参数（`args[0]`）、`nameNotRequired=true`、走 `rule.go:201` 收集。
- 像 `requiredWith`：是 `*Validation` 方法、入 `ctxValidatorBuilders` + `callValidator` switch。

实现时**重点核对参数个数校验**（`valueValidate` 里 `if r.nameNotRequired { fm.checkArgNum(...) }`）：`rule_one_of` 是 `nameNotRequired=true`，会走 `checkArgNum`。需确保它对"1 个列表参数 + 方法签名"不误报——直接对照 `enum` 当前如何通过该检查来对齐（enum 也是 nameNotRequired、1 列表参数）。这是本特性**唯一需要小心的接线点**。

## 4. 范围与边界（phase 1）

1. **仅支持无参子校验器**（`ip`/`cidr`/`email`/`url`/`alpha`/…）。**带参子规则**（如 `min:5`）**暂不支持**：逗号既分隔子项名、又是校验器自身参数分隔符，`rule_one_of:min:5,max:10` 无法无歧义解析。留后续（需换分隔符或嵌套语法，见 §6）。
2. **未知子校验器名**的处理（**需决策**）：
   - **A. build 期 fail-fast（推荐）**：解析时校验每个名字存在性，不存在则 `panic`（与现有"未知 validator 直接 panic"一致，早暴露拼写错误）。
   - B. 运行期该子项视为 fail（`continue`）：更宽容但会把拼写错误悄悄吞掉。
3. **空列表 / 单元素**：`rule_one_of:ip` 退化为等价 `ip`；空列表应 fail 或 panic（建议 build 期报错）。

## 5. 与 required / SkipOnEmpty 交互

`rule_one_of` 是普通（非 required）校验器：
- `SkipOnEmpty=true`（默认）下，**空值被跳过**（可选字段，不校验）——与所有非必填校验器一致。
- 要"必填且满足其一"：`required|rule_one_of:ip,cidr`（`required` 管非空，`rule_one_of` 管格式其一）。

## 6. 不在本设计内（后续/另议）

- **逻辑"非"**：`not:xxx`（取反单规则）——可另开小特性。
- **逻辑"与组"/嵌套**：`|` 已是与；如需显式 `rule_all_of` 或嵌套括号表达式，属更大的规则表达式引擎，单独立项。
- **带参子规则**：见 §4.1。

## 7. 回归风险

- **纯新增**一个校验器，不改任何现有 validator 行为 → golden `TestRuleCompat` 预期逐字节不变。
- 改动点集中（5 处、~50–80 行），无跨模块扩散。
- 必跑：golden ×3 + 全量 + race + vet。

## 8. 测试计划（`TestIssue_292_v2`，放 `issues_x3_test.go`）

- `ip` 命中 → 通过；`cidr` 命中 → 通过；两者都不满足 → 失败（断言错误消息含 "did not satisfy any"）。
- tag 写法 `validate:"rule_one_of:ip,cidr"` 与程序化 `StringRule` 各一例。
- 别名解析：`ip`/`IP`/`cidr`/`CIDR` 都能命中。
- 与 `required` 组合：空值在 `required|rule_one_of:...` 下被 required 拒；非空非法被 rule_one_of 拒；合法通过。
- 仅 `rule_one_of`（无 required）+ 空值 → SkipOnEmpty 跳过、通过（语义说明）。
- （决策 A 时）未知子校验器名触发 panic 的用例。

## 11. 实施记录（与设计稿的两处差异）

实际落地与 §3.2 代码片段有两处差异（按已拍板决策调整）：

1. **方法签名**：实际为 `func (v *Validation) RuleOneOf(val any, rules any) bool`（**2 入参**，不含 `field`）。
   这是为了**镜像 `Enum(val, enum any)` 的形状**：绑定方法 `numIn=2`，`checkArgNum(1 列表参数 + addNum=1 = 2)` 恰好通过，与 `enum` 完全对齐。phase1 子校验器无参，不需要 `field`，调用时传 `callValidator(v, fm, "", val, nil, 1, nil)`。
2. **未知子校验器**：实际 `panic`（决策 A），而非 §3.2 片段里保守的 `continue`。空列表/非 `[]string` 参数同样 `panic`。
3. **接线位置**：`rule_one_of` 的 `ctxValidatorBuilders` 入口在 `validate.go` 的 `init()` 中注册（而非 map 字面量内），以**打破静态初始化环**（`RuleOneOf → validatorMeta → ctxValidatorBuilders`）。其余 required-family 方法体不引用 `validatorMeta`，故无此环。

实际 5 处接线：`rule.go:201`（enum 范式加 case）、`validators.go`（`RuleOneOf` 方法，紧邻 `Required` 后）、`validate.go init()`（ctx 绑定）、`validating.go` `callValidator` switch（enum 旁）、`messages.go`（builtin 文案，enum 旁）。
